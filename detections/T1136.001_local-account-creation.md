# Local Account Creation Detection

## MITRE ATT&CK

| Field | Value |
|-------|-------|
| **Technique** | [T1136.001 — Create Account: Local Account](https://attack.mitre.org/techniques/T1136/001/) |
| **Tactic** | Persistence |
| **Severity** | Medium-High |
| **Data Source** | Windows Security Event Log, Sysmon |
| **Key Event ID** | 4720 (User Account Created), Sysmon EID 1 (Process Create for net.exe/net1.exe) |

---

## Description

After gaining initial access, attackers frequently create local accounts to maintain persistence on a compromised system. A new local admin account gives the attacker a backdoor that survives password resets on the original compromised account.

This technique is categorized under the **Persistence** tactic because the goal is to maintain long-term access to the system, even if the initial access vector is discovered and remediated.

---

## Attack Simulation

**Tool:** Atomic Red Team (T1136.001)
**Target:** Target-PC (192.168.10.100) — Admin PowerShell

```powershell
# Import the module (required each session)
Import-Module "C:\AtomicRedTeam\invoke-atomicredteam\Invoke-AtomicRedTeam.psd1"

# Execute the test — creates a local user account
Invoke-AtomicTest T1136.001
```

Atomic Red Team's T1136.001 test creates a local user account using `net user /add`, which triggers:
- Security Event ID 4720 (A user account was created)
- Sysmon Event ID 1 (Process Creation for net.exe or net1.exe)

### Cleanup
```powershell
# Remove the test account after detection validation
Invoke-AtomicTest T1136.001 -Cleanup
```

## Splunk Detection Query

### Primary Detection — New Account Created

```spl
index=endpoint EventCode=4720 host="target-PC"
| table _time, user, Account_Name, Security_ID
| sort -_time
```

Alerts on any new local account creation on the Target-PC.

### Process-Based Detection — net.exe Usage

```spl
index=endpoint source="XmlWinEventLog:Microsoft-Windows-Sysmon/Operational" EventCode=1
  (Image="*\\net.exe" OR Image="*\\net1.exe")
  CommandLine="*user*" CommandLine="*/add*"
| table _time, Computer, User, Image, CommandLine, ParentImage
| sort -_time
```

Detects the actual command used to create the account. This is valuable because it captures the full command line, revealing the username and any flags used.

## Expected Results

When `Invoke-AtomicTest T1136.001` runs:
- **1+ events** with EventCode=4720 showing the new account name
- **1+ events** in Sysmon (EID 1) showing `net.exe` or `net1.exe` with `/add` in the command line
- The `ParentImage` field in Sysmon reveals the process that initiated the account creation (PowerShell in this case)

---

## Analyst Triage Steps

1. **Identify the new account.** What is the username? Does it follow the organization's naming convention? Unfamiliar names like `backdoor`, `test`, or random strings are suspicious.

2. **Check who created it.** The 4720 event shows the `Subject` — which account performed the action. Was it the domain admin during routine operations, or an unexpected account?

3. **Check the parent process.** In the Sysmon event, what process spawned `net.exe`? If the parent is `powershell.exe` or `cmd.exe` launched from an unusual location, this is suspicious.

4. **Check if the account was added to privileged groups.** Search for EventCode=4732 (Member Added to Security-Enabled Local Group) for the same timeframe. Attackers often add new accounts to the Administrators group.

5. **Check for subsequent logon.** Search for EventCode=4624 with the new account name. If the newly created account logged in, the attacker is actively using it.

6. **Determine if it was authorized.** Correlate with change management records. Was a new account creation approved?

7. **Escalate if unauthorized.** Disable the account, investigate the system for additional indicators of compromise, and check for lateral movement.

---

## False Positive Considerations

| Scenario | How to Distinguish |
|----------|-------------------|
| IT admin creating a service account | Verify against change management. Admin accounts creating accounts during business hours is expected. |
| Software installation creating a local account | Some applications create service accounts during install. Check the parent process — it should be an installer. |
| Automated provisioning scripts | Should originate from known automation tools with expected account naming patterns. |
