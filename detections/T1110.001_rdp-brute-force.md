# RDP Brute Force Detection

## MITRE ATT&CK

| Field | Value |
|-------|-------|
| **Technique** | [T1110.001 — Brute Force: Password Guessing](https://attack.mitre.org/techniques/T1110/001/) |
| **Tactic** | Credential Access |
| **Severity** | High |
| **Data Source** | Windows Security Event Log |
| **Key Event ID** | 4625 (Failed Logon), 4624 (Successful Logon) |

---

## Description

An attacker attempts to gain access to a domain account by systematically trying common passwords against the RDP service (port 3389). This technique is frequently used in real-world attacks against internet-facing RDP servers and is one of the most common initial access vectors.

A successful brute force attack produces a distinctive pattern: a rapid burst of failed logon events (4625) followed by a single successful logon (4624) for the same account from the same source IP.

---

## Attack Simulation

**Source:** Kali Linux (192.168.10.250)
**Target:** Target-PC (192.168.10.100) — RDP on port 3389
**Tool:** xfreerdp (FreeRDP 3.x)

```bash
# Brute force RDP login for user tsmith using a password wordlist
while IFS= read -r pass; do
  echo "Trying: $pass"
  timeout 10 xfreerdp /v:192.168.10.100 /u:tsmith /p:"$pass" /cert:ignore +auth-only 2>&1 |
  grep -q "exit status 1" && echo ">>> SUCCESS: $pass <<<" && break
done < ~/Desktop/ad-project/passwords.txt
```

The wordlist contains 21 common passwords. The correct password is the last entry, meaning 20 failed attempts precede the successful one.

> **Note:** Crowbar 0.4.2 does not work with FreeRDP 3.x due to reversed exit codes. See [docs/lessons-learned.md](../docs/lessons-learned.md) for details.

## Splunk Detection Query

### Primary Detection — Failed Logon Spike

```spl
index=endpoint EventCode=4625
| stats count by src_ip, dest, user
| where count > 10
| sort -count
```

This query identifies source IPs that have generated more than 10 failed logon attempts, grouped by target machine and user account. A threshold of 10 minimizes false positives while catching brute force patterns.

### Correlated Detection — Brute Force Followed by Success

```spl
index=endpoint EventCode IN (4625, 4624) user=tsmith
| sort _time
| stats count(eval(EventCode=4625)) as failures, count(eval(EventCode=4624)) as successes by user, src_ip
| where failures > 5 AND successes > 0
```

This query is more precise: it finds accounts where multiple failures were followed by a successful logon from the same source — the signature of a successful brute force.

## Expected Results

When the brute force script runs with 21 passwords:
- **20 events** with EventCode=4625 (Failed Logon) appear in Splunk
- **1 event** with EventCode=4624 (Successful Logon) for the correct password
- All events originate from the same source IP (Kali: 192.168.10.250)
- Events appear within a 1-2 minute window

---

## Analyst Triage Steps

As a SOC analyst investigating this alert, follow these steps:

1. **Confirm the pattern.** Query for all 4625 events for the affected account in the alert timeframe. Is there a burst of failures in a short window?

2. **Check for successful logon.** Search for EventCode=4624 for the same account immediately after the failures. A success after multiple failures = likely compromise.

3. **Identify the source.** What is the source IP? Is it internal or external? Does it belong to a known system or user?

4. **Check for account lockout.** Search for EventCode=4740 (Account Lockout). If the account was locked, the attack may have been partially mitigated.

5. **Check for lateral movement.** If a successful logon occurred, search for subsequent activity from that account: new processes (Sysmon EID 1), network connections (Sysmon EID 3), or logons to other machines (4624 on other hosts).

6. **Determine scope.** Was only one account targeted, or were multiple accounts attacked? Search for 4625 events across all users from the same source IP.

7. **Escalate.** If a successful logon followed the brute force:
   - Disable the compromised account
   - Isolate the target machine
   - Initiate incident response procedures

## False Positive Considerations

| Scenario | How to Distinguish |
|----------|-------------------|
| User forgot password and tried multiple times | Typically 3-5 attempts, not 10+. Check if the user reported a password issue. |
| Service account with expired credentials | Failures will repeat at regular intervals (e.g., every 5 minutes). Check the account type. |
| Help desk resetting passwords | Correlate with ticketing system. Failures should coincide with a reset request. |
| VPN or SSO misconfiguration | Failures from an internal gateway IP, affecting multiple users simultaneously. |
